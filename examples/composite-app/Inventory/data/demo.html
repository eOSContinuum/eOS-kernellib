composite-app: browser-real WebAuthn

Principal and agents. YOU are the principal -- the party on whose behalf everything here runs. Your passkeys AUTHENTICATE you (several can prove the same platform identity, and recovery codes are the fallback -- losing one authenticator never loses the identity). What you may DO is authorized separately: an operator grants capabilities to your identity, and step 7 mints agents to which you can delegate a subset of what you hold (the API's routes and console verbs call the principal the agent's "controller"). The page holds one session at a time; the banner above the log always says who is acting, and buttons that need a different session disable themselves and say why. One wire note: API responses name the ACTING identity's subject string in a field called principal -- even for an agent's actions; this page keeps "principal" for you, the party agents act for.
Two planes. This page is the application plane; grants and delegable flags belong to the operator console. The demo deploy pre-provisions the only operator actions the walk needs (a delegable demo capability, granted to the principal at registration); step 6 keeps one refusal alive on purpose so you see the boundary.
The walk: 1a register → 3a create → 5 audit → 6 wipe (403) → 14 report (200) → 7 mint → 8 list → 13 agent login → 14 report (403) → 7 mint as the agent (403) → 1b back → 11 delegate → 13 agent login → 14 report (200, delegated) → 3a create as the agent → 3b update the principal's item (403) → 1b back → 9/10 suspend & resume → 15 recovery kit → 1c recover → 16 list passkeys → 17 revoke the old one → 18 enroll a fresh one → 16 list again (two, no recovery code spent). Streams are live the whole time; the heartbeat pinned under the log is the server pushing over SSE every 10 seconds.

1. Ways in (unauthenticated -- pick by what you hold)

Every path starts logged out; the three entries differ only in what you hold. Nothing yet: 1a creates a NEW platform identity whose first credential is this device's passkey (the demo grants it example:delegation-demo at registration). A passkey on this device: 1b logs in -- also the way back from an agent session. A recovery kit (uuid + one code, provisioned at 15): 1c enrolls a fresh passkey on THIS device and atomically binds it to your EXISTING identity, spending the code -- recovery never logs in with an old passkey. 1c serves the new-device case (the old passkey keeps working: two devices, one identity) and the lost-device case: recover here, then list (16) and revoke (17) the lost device's passkey. The uuid is the demo's account handle -- the platform keys identities by uuid, the passkey sheet's demo@localhost label is decorative, and a real app would layer a verified handle-to-identity directory on top; this page remembers your principal uuid in this browser and prefills the field.

Session

2 revokes the current session, whoever it belongs to; the banner always names the acting principal.

Items and audit (any session; reads are public)

3a mutates as WHOEVER is logged in -- watch the audit trail and the live stream attribute it to the principal or the agent. 3b is the APPLICATION-TIER refusal: only an item's CREATOR may update it, so the agent cannot touch the principal's items even while delegated -- create as the principal, switch to the agent (13), then 3b on that item answers 403 "not the item's creator" (3a fills the id field with the last created item). 4 and 5 are public reads, no session needed. 6 is gated by example:inventory-admin, which nobody in this demo holds: expect 403 from the capability check -- in production an operator grants it on the console (identity grant <uuid> example:inventory-admin).

Agents and delegation (management needs the principal's session)

7 mints an agent -- the response is the only time the token plaintext exists, and the page fills it into step 13's field. 7 stays enabled under an AGENT session as the standing lesson: it refuses (only human identities control agents), because delegation grants capabilities, never the principal's own standing -- an agent cannot mint, manage, or re-delegate. 8 lists your agents with suspension state and delegated capabilities. 9/10 suspend and resume the agent named in the uuid field (suspend also revokes its live sessions). 11 delegates the capability in the field to that agent: example:delegation-demo succeeds because the demo deploy granted it to you at registration and flagged it delegable -- try example:inventory-admin to see the refusal an unheld, non-delegable capability gets. 12 withdraws the delegation. 13 trades the minted token for the AGENT's session. 14 makes the delegation observable: the report is gated by example:delegation-demo, so it answers 200 as the principal, 403 as the agent until 11 delegates, 200 while the delegation stands, and 403 again after 12 -- while wipe (6) stays 403 for both, a power neither was granted. Watch step 8 or the live agent stream reflect each change.

Recovery kit and passkeys (principal session only)

15 provisions recovery codes for the PRINCIPAL while its session is live -- the response is the only time the plaintext exists, and the log prints the kit (uuid + codes) to save as one unit: it is exactly what 1c needs on a new device. 16 lists the passkeys bound to your identity (bookkeeping only, never key material; the one this browser holds is marked) and fills the field with the oldest OTHER one -- after a 1c recovery, that is the lost device's. 17 revokes the passkey in the field: the lost-device story's second half. The facade refuses to revoke your LAST passkey, so a fresh identity with a single passkey answers 403 until another is bound. 18 binds that other one the routine way: ENROLLMENT -- a fresh passkey on your logged-in identity, no recovery code spent; this is how a second device joins, and recovery (1c) goes back to being the loss path only. All four disable under an agent session; passkey login (1b) is the way back.
